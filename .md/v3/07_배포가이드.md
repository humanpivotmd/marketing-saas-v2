# v3 배포 가이드

## 1. Railway 배포

- 자동 배포: GitHub 연동, main 브랜치 push 시 자동 빌드/배포
- 빌드 명령: `npm run build` (Next.js)
- 시작 명령: `npm start`
- 포트: Railway에서 자동 할당 (PORT 환경변수)

## 2. Supabase 마이그레이션

### 마이그레이션 파일
- `supabase/migrations/001_init.sql` - 초기 스키마 (v1~v2)
- `supabase/migrations/002_seed.sql` - 시드 데이터
- `supabase/migrations/003_v3_pipeline.sql` - v3 신규 테이블 + 기존 변경
- `supabase/migrations/004_add_core_message.sql` - projects.core_message 추가
- `supabase/migrations/005_add_missing_columns.sql` - contents 누락 컬럼 추가

### 실행 방법
Supabase Dashboard의 SQL Editor 또는 `run_migration` RPC를 통해 실행:
```sql
-- 003_v3_pipeline.sql 내용을 SQL Editor에 붙여넣기 후 실행
-- 또는 Supabase CLI:
-- supabase db push
```

주의사항:
- `IF NOT EXISTS` / `IF NOT EXISTS (SELECT 1 FROM pg_policies ...)` 패턴으로 멱등성 보장
- 기존 CHECK 제약조건은 DROP 후 재생성 (`ALTER TABLE ... DROP CONSTRAINT IF EXISTS ... ADD CONSTRAINT ...`)
- Seed 데이터(industries 14개, plans 2개)는 `ON CONFLICT DO NOTHING` / `WHERE NOT EXISTS`로 중복 방지

## 4. 테스트 계정

초기 seed 데이터 기반. 002_seed.sql에서 생성된 계정 사용:
- 실제 계정 정보는 Supabase Dashboard의 users 테이블에서 확인
- 관리자 계정: role = 'admin' 또는 'super_admin'
- 일반 계정: role = 'user'
- 프리미엄 테스트: plans 테이블의 premium 플랜을 유저에게 할당

## 5. 배포 전 확인사항

2. ANTHROPIC_API_KEY 또는 CLAUDE_API_KEY 환경변수 설정 (Railway에는 `CLAUDE_API_KEY`로 등록)

4. plans 테이블에 starter/premium 플랜 존재 확인

6. **프로덕션 DB 컬럼과 코드 일치 확인** — 특히 contents 테이블 (channel, metadata 등)
