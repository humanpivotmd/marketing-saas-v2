## 1. QA팀 (현재 82점)

### 이슈 1: Supabase DNS 에러 — 다른 프로젝트 URL 호출
- **사용자 영향도**: 높음 — 특정 기능 호출 시 서버 에러가 발생하면 사용자는 "서비스가 고장났다"고 인식
- **구현 난이도**: 쉬움
- **해결 방안**: 환경변수(`SUPABASE_URL`, `SUPABASE_ANON_KEY`)를 Railway 배포 환경에서 현재 프로젝트의 올바른 Supabase 프로젝트로 교체. `src/lib/supabase/server.ts`, `client.ts`에서 하드코딩된 URL이 없는지 추가 확인. 배포 후 `/api/auth/me` 등 주요 엔드포인트에서 DNS 정상 응답 테스트.

### 이슈 2: settings 이메일 인증 리다이렉트 비일관성
- **사용자 영향도**: 중간 — 이메일 인증 후 엉뚱한 페이지로 이동하면 사용자가 혼란
- **구현 난이도**: 쉬움
- **해결 방안**: `/api/auth/verify-email` 라우트의 리다이렉트 URL을 settings 프로필 탭(`/settings#profile`)으로 통일. 회원가입 시 인증 메일의 리다이렉트는 `/dashboard`로, 설정에서 이메일 변경 시 인증 메일의 리다이렉트는 `/settings#profile`로 분기 처리.

## 2. UX팀

### 이슈 1: 콘텐츠 유형(블로그/Threads/인스타) 선택이 1단계에서 안 보임
- **사용자 영향도**: 높음 — 현재 `/contents/new` 1단계에서 키워드만 선택하고 채널 선택(`selectedChannels`)은 코드상 기본값 `['blog']`로 숨겨져 있음. 자영업자는 "인스타 글 만들려고 왔는데 어디서 선택하지?" 상태
- **구현 난이도**: 보통
- **해결 방안**: Step 1에서 키워드 선택 바로 아래에 채널 선택 UI를 노출. CHANNELS 배열(블로그/Threads/인스타그램/영상 스크립트)을 칩(chip) 형태로 표시하고, 최소 1개 선택 필수 안내. "어떤 채널에 올릴 콘텐츠인가요?"라는 질문형 레이블 추가.

### 이슈 2: "다음" 버튼 비활성 이유 안내 없음
- **사용자 영향도**: 높음 — 버튼이 비활성인데 왜 누를 수 없는지 모르면 이탈
- **구현 난이도**: 쉬움
- **해결 방안**: 비활성 버튼 아래에 인라인 안내 텍스트 표시. 예: "키워드를 선택하거나 입력해주세요", "채널을 1개 이상 선택해주세요". disabled 상태일 때 `text-text-tertiary text-xs mt-1`로 조건별 메시지 렌더링.

### 이슈 3: 등급 "C"의 의미 설명 부족
- **사용자 영향도**: 높음 — 키워드 등급이 A+~D-까지 12단계인데, "C등급이면 좋은 건지 나쁜 건지" 직관적으로 모름
- **구현 난이도**: 쉬움
- **해결 방안**: 키워드 등급 옆에 한 줄 설명을 항상 표시. 현재 `GRADE_MAP`에 `difficulty`("높음")와 `opportunity`("낮음") 값이 이미 있으므로, 등급 Badge 옆에 "경쟁: 높음 / 기회: 낮음" 형태로 표시. 등급 영역 클릭 시 전체 등급표를 모달/팝오버로 보여주는 "등급 안내" 링크 추가.

## 3. 모바일팀 (현재 68점)

### 이슈 1: 설정 탭 5개가 375px에서 가로 잘림
- **사용자 영향도**: 높음 — 모바일에서 설정 페이지의 '알림' 탭이 보이지 않으면 해당 기능에 접근 불가
- **구현 난이도**: 쉬움
- **해결 방안**: 현재 settings 탭 컨테이너에 `overflow-x-auto scrollbar-hide`가 있지만, 스크롤 가능하다는 시각적 힌트가 없음. 두 가지 개선 필요:
  1. 탭을 375px 기준으로 `text-xs`로 축소하거나 아이콘+텍스트 조합으로 변경
  2. 우측에 그라디언트 페이드 + 화살표 아이콘으로 "더 있음" 표시
  3. 대안: 모바일에서는 드롭다운 셀렉트로 전환

### 이슈 2: 캘린더 "예약 등록" 버튼 우측 잘림
- **사용자 영향도**: 중간 — 예약 등록을 못하면 핵심 기능(자동 포스팅) 사용 불가
- **구현 난이도**: 쉬움
- **해결 방안**: 캘린더 페이지(`/calendar`)의 상단 버튼 영역을 모바일에서 `flex-wrap` 처리하거나, 버튼을 `w-full`로 변경. 또는 플로팅 액션 버튼(FAB) 패턴으로 우하단 고정.

## 4. 접근성팀 (현재 72점)

### 이슈 1: settings 탭에 role="tab" ARIA 없음
- **사용자 영향도**: 중간 — 스크린 리더 사용자가 탭 내비게이션을 인식 불가
- **구현 난이도**: 쉬움
- **해결 방안**: settings 탭 컨테이너에 `role="tablist"`, 각 탭 버튼에 `role="tab"`, `aria-selected={activeTab === tab.id}`, `aria-controls={`panel-${tab.id}`}` 추가. 탭 콘텐츠 영역에 `role="tabpanel"`, `id={`panel-${tab.id}`}`, `aria-labelledby={`tab-${tab.id}`}` 추가. 좌우 화살표 키로 탭 이동 키보드 핸들러 구현.

### 이슈 2: contents 필터에 aria-pressed 없음
- **사용자 영향도**: 낮음 — 시각 장애인 사용자 대상이나 법적 접근성 준수에 필요
- **구현 난이도**: 쉬움
- **해결 방안**: 콘텐츠 목록 필터 버튼(전체/블로그/인스타그램 등)에 `aria-pressed={isActive}` 속성 추가.

### 이슈 3: 폼 에러 메시지에 aria-live 없음
- **사용자 영향도**: 중간 — 입력 오류 시 스크린 리더가 에러를 읽어주지 않으면 사용자가 무엇이 잘못됐는지 모름
- **구현 난이도**: 쉬움
- **해결 방안**: Input 컴포넌트(`src/components/ui/Input.tsx`)의 에러 메시지 영역에 `role="alert"` 또는 `aria-live="polite"` 추가. Toast 컴포넌트에도 `role="status"` 또는 `aria-live="assertive"` 확인.

## 5. 보안팀 (현재 72점)

### 이슈 1: CSP에 unsafe-inline 허용
- **사용자 영향도**: 낮음 (직접 체감 없으나, 공격 시 피해 심각)
- **구현 난이도**: 보통
- **해결 방안**: Next.js의 `nonce` 기반 CSP로 전환. `next.config.js`에서 CSP 헤더를 설정하되, `script-src 'nonce-{random}'`으로 교체. 인라인 스크립트(JSON-LD 등)에 nonce 속성 부여. styled-jsx나 인라인 스타일은 `style-src 'nonce-{random}'`으로 처리.

### 이슈 2: CSRF 보호 부재
- **사용자 영향도**: 낮음 (직접 체감 없으나, 공격 시 데이터 변조 위험)
- **구현 난이도**: 보통
- **해결 방안**: 상태 변경 API(POST/PUT/DELETE)에 CSRF 토큰 검증 추가. 방법: (1) Double Submit Cookie 패턴 — 서버가 `csrf-token` 쿠키를 설정하고, 클라이언트가 요청 헤더 `X-CSRF-Token`에 같은 값을 전송, 서버에서 일치 여부 확인. (2) SameSite=Strict 쿠키 속성 추가로 기본 방어. 현재 JWT가 sessionStorage에 있으므로 CSRF 위험은 쿠키 기반 인증보다 낮지만, HttpOnly 쿠키로 전환했다면 CSRF 보호 필수.

### 이슈 3: JWT payload에 email, role 등 민감정보
- **사용자 영향도**: 낮음 (일반 사용자는 모르지만, 토큰 탈취 시 정보 노출)
- **구현 난이도**: 보통
- **해결 방안**: JWT payload에서 `email` 제거, `role`은 서버 사이드에서만 검증. payload를 `{ sub: userId, iat, exp }` 최소화하고, 사용자 정보는 `/api/auth/me` API로만 제공. 기존 토큰과의 하위 호환을 위해 마이그레이션 기간(2주) 동안 두 형식 모두 허용.

## 6. 성능팀 (현재 68점)

### 이슈 1: Pretendard 폰트 서브셋 8~13개 매회 로드 (200KB+)
- **사용자 영향도**: 높음 — 모바일 3G/LTE 환경에서 초기 로딩이 느려 이탈 유발. 자영업자는 매장에서 빠르게 확인하려는 패턴이 많음
- **구현 난이도**: 보통
- **해결 방안**:
  1. `next/font/local`로 Pretendard를 로컬 번들링하여 subset 최적화 (Regular, Bold 2개 웨이트만)
  2. `font-display: swap` 설정으로 FOIT 방지
  3. 또는 `@fontsource/pretendard`로 필요한 웨이트만 import
  4. 목표: 폰트 로드 50KB 이하

### 이슈 2: admin API 불필요 호출 (일반 사용자)
- **사용자 영향도**: 중간 — 직접 체감은 적으나 불필요한 네트워크 요청이 페이지 로드 지연
- **구현 난이도**: 쉬움
- **해결 방안**: 대시보드 레이아웃이나 사이드바에서 admin API(`/api/admin/*`)를 조건 없이 호출하는 부분을 찾아, `role === 'admin'` 체크 후에만 호출하도록 수정. `useEffect` 내 조건부 fetch로 변경.

## 7. 카피라이터팀 (현재 72점)

### 이슈 1: "API 접근" → 비전문가에게 생소
- **사용자 영향도**: 중간 — Business 플랜 피처 목록에서 "API 접근"이 무엇인지 자영업자는 모름. 가치 전달 실패
- **구현 난이도**: 쉬움
- **해결 방안**: 요금제 페이지(`/pricing`)와 랜딩 페이지에서:
  - "API 접근" → "외부 서비스 연동" 또는 "자동화 연동"
  - 하위 설명 추가: "쇼핑몰, POS 등 기존 시스템과 자동 연결"
  - `FEATURES` 배열의 `label: 'API 접근'` → `label: '외부 서비스 연동'`으로 수정

### 이슈 2: Business CTA "문의하기"가 /login으로 연결
- **사용자 영향도**: 높음 — "문의하기"를 누르면 로그인 페이지가 나오는 것은 사용자 기대에 완전히 어긋남. 결제/문의 의향이 있는 고가치 리드를 잃음
- **구현 난이도**: 쉬움
- **해결 방안**: 두 가지 중 선택:
  1. **즉시**: "문의하기" CTA를 `mailto:support@marketingflow.co.kr` 또는 카카오톡 채널 링크로 변경
  2. **단기**: `/support` 페이지로 연결하여 문의 폼 제공 (이미 `/support` 페이지 존재)
  - 랜딩 페이지의 Business 플랜 CTA href를 `/support`로 변경

## 8. 기획팀 (현재 72점)

### 이슈 1: 토스페이먼츠 결제 미연동
- **사용자 영향도**: 높음 — 유료 전환이 불가능하면 매출 제로. 사용자가 "결제하기"를 눌러도 실제 결제가 안 됨
- **구현 난이도**: 어려움
- **해결 방안**:
  1. 토스페이먼츠 SDK 연동 (`/pricing/checkout` 페이지 기반)
  2. 결제 플로우: 플랜 선택 → 토스 결제창 → 성공 콜백 → `/api/payments/checkout`에서 결제 검증 → DB 플랜 업그레이드
  3. 정기결제(빌링키) 방식으로 월간/연간 자동 갱신
  4. 환불 정책 및 구독 취소 로직(`/api/subscriptions/cancel` 이미 존재) 연동
  5. 최소 MVP: 수동 계좌이체 안내 + 관리자 수동 플랜 변경으로 임시 운영 가능

### 이슈 2: 사업체 정보 입력 기능 없음 (자영업자 핵심 니즈)
- **사용자 영향도**: 높음 — "내 카페 이름, 주소, 업종"을 입력하면 그에 맞는 콘텐츠가 자동 생성되는 것이 핵심 가치. 현재는 Brand Voice 프리셋(카페/음식점, 뷰티/패션 등)만 있고, 실제 사업체 정보를 구조화하여 입력하는 UI가 없음
- **구현 난이도**: 보통
- **해결 방안**:
  1. 온보딩(`/onboarding`) 또는 settings 프로필 탭에 "내 사업체 정보" 섹션 추가
  2. 입력 필드: 상호명, 업종(드롭다운), 주소, 주요 상품/서비스, 타겟 고객층
  3. DB: `users` 테이블에 `business_name`, `business_type`, `business_address`, `business_description` 컬럼 추가 또는 별도 `businesses` 테이블
  4. 콘텐츠 생성 시 사업체 정보를 프롬프트에 자동 주입 → 개인화된 마케팅 콘텐츠 생성

## 9. 사용자 테스트팀 (현재 52점)

### 이슈 1: 초보 사용자가 "Brand Voice" 의미 모름
- **사용자 영향도**: 높음 — settings의 "Brand Voice" 탭과 콘텐츠 생성 시 Brand Voice 선택이 핵심 기능인데, 용어 자체가 장벽
- **구현 난이도**: 쉬움
- **해결 방안**:
  - "Brand Voice" → "우리 가게 말투" 또는 "브랜드 말투 설정"으로 한글화
  - settings 탭 레이블: `'Brand Voice'` → `'브랜드 말투'`
  - 최초 진입 시 한 줄 설명 표시: "AI가 우리 가게 스타일에 맞는 글을 쓸 수 있도록 말투를 설정하세요"
  - 프리셋(카페/음식점, 뷰티/패션 등)을 더 강조하여 "선택만 하면 끝"이라는 인식 제공

### 이슈 2: 키워드 등급 A+~D- 의미 안내 없음
- **사용자 영향도**: 높음 — 키워드 분석 결과에서 등급만 보여주고 "그래서 뭘 해야 하는지" 모름
- **구현 난이도**: 쉬움
- **해결 방안**:
  - 키워드 목록/상세 페이지에 등급별 행동 가이드 추가:
    - A+~A-: "지금 바로 글을 쓰세요! 경쟁이 낮아 상위 노출 가능성이 높습니다"
    - B+~B-: "충분히 도전할 만합니다. 꾸준히 글을 올리면 효과를 볼 수 있습니다"
    - C+~C: "경쟁이 치열합니다. 차별화된 콘텐츠가 필요합니다"
    - C-~D-: "다른 키워드를 추천합니다. 상위 노출이 매우 어렵습니다"
  - 등급 Badge에 tooltip 또는 바로 옆에 한 줄 요약 표시

### 이슈 3: "SEO 점수" 설명 없음

## 회의 결론

1. **Phase 1 (즉시)**: 텍스트 변경과 환경 설정 수정만으로 해결 가능한 12개 항목. 코드 변경이 적고 사용자 체감 효과가 큼. 특히 용어 한글화와 안내 텍스트 추가가 사용자 테스트 점수를 16점 올릴 수 있음.

2. **Phase 2 (1주)**: UI 구조 변경이 필요한 8개 항목. 콘텐츠 생성 플로우 개선과 폰트 최적화가 핵심. 이 단계까지 완료하면 전체 평균 85점 이상 달성.

3. **Phase 3 (1개월)**: 사업체 정보와 결제 연동은 서비스의 핵심 비즈니스 가치와 직결. 이것이 완료되어야 실제 유료 전환과 자영업자 리텐션이 가능.
