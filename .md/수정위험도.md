## 수정 유형별 처리 순서

### DB 컬럼 추가/변경/삭제 🔴
```
1. supabase/migrations/ 새 파일 작성
2. src/types/index.ts 타입 업데이트
3. 관련 API route 수정
4. 프론트 호출부 수정
5. npm run build 확인
→ 순서 어기면 TypeScript 에러 폭발
```

### 프롬프트 수정 🟡
```
1. src/lib/prompts/pipeline.ts 확인
2. PipelineContext 변경 필요 여부 파악
3. 영향받는 generate API 목록 확인
4. 수정 후 해당 STEP 테스트
→ B2B/B2C 분기 양쪽 다 확인 필수
```

### API Route 수정 🟡
```
1. 해당 API를 호출하는 프론트 파일 grep으로 찾기
2. 요청/응답 형식 변경 여부 확인
3. 형식 변경 시 프론트 호출부 동시 수정
4. src/types/index.ts 타입 변경 필요 여부 확인
```

### 상수 추가/변경 🔴→🟡
```
새 상수 추가: 비교적 안전 (단, 기존 하드코딩 grep 확인)
기존 상수 이름 변경: 🔴 — 사용처 전부 grep 후 동시 변경
기존 상수 값 변경: 🟡 — 영향 범위 파악 후 진행
```

## 자주 꼬이는 패턴 & 방지법

### ❌ 패턴 1: projects 수정 → create/* 전체 깨짐
- 원인: projects API 응답 형식 or settings_snapshot 구조 변경
- 방지: projects 관련 수정 시 create/* 5개 페이지 + FlowGuard 전부 확인

### ❌ 패턴 2: DB 컬럼 추가 → TypeScript 에러 폭발
- 원인: types/index.ts 업데이트 없이 API만 수정
- 방지: DB 변경 = types 변경 세트로 반드시 함께

### ❌ 패턴 3: 프롬프트 수정 → 다른 STEP 결과 이상해짐
- 원인: pipeline.ts의 PipelineContext를 여러 함수가 공유
- 방지: 어떤 buildXxxPrompt 함수가 영향받는지 먼저 파악

### ❌ 패턴 4: 상수 이름 변경 → 일부 페이지만 업데이트
- 원인: grep 없이 눈에 보이는 곳만 수정
- 방지: 상수 이름 변경 = 반드시 grep으로 전수 확인

### ❌ 패턴 5: 공통 훅 수정 → 16개 페이지 중 일부만 깨짐
- 원인: useAsyncAction.ts 인터페이스 변경
- 방지: 훅 수정 시 16개 페이지 전부 확인 (npm run build로 검증)
