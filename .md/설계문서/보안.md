## 1. 인증 (Authentication)

### JWT
- 라이브러리: `jsonwebtoken`, 알고리즘 HS256
- 페이로드: `{ id, userId, role }` (최소화 원칙)
- 만료: 24시간
- 전송: `Authorization: Bearer {token}` 헤더 + HttpOnly 쿠키

### 쿠키 설정
```
httpOnly: true
secure: production만
sameSite: 'lax'
maxAge: 24시간
path: '/'
```

### 비밀번호
- bcrypt 12 rounds
- 최소 요건: 8자 이상 + 영문 + 숫자 (Zod 검증)
- 변경 시 현재 비밀번호 확인 필수

### 이메일 인증 토큰
- 생성: `crypto.randomUUID()`
- 인증 링크: 24시간 만료
- 재설정 링크: 1시간 만료

### 이메일 열거 방지
- forgot-password에서 유저 존재 여부와 무관하게 항상 동일한 성공 응답 반환

코드 참조: `src/lib/auth.ts`

## 2. 인가 (Authorization)

### RBAC (역할 기반 접근 제어)

| 역할 | 접근 범위 |
|------|----------|
| `user` | 본인 데이터만 |
| `member` | 본인 데이터만 |
| `team_admin` | 팀 데이터 |
| `admin` | 전체 데이터 + 관리 기능 + 사용 제한 해제 |
| `super_admin` | admin과 동일 |

### RLS (Row Level Security)

모든 21개 테이블에 RLS 활성화. 상세 정책은 `DB스키마.md` RLS 절 참조.

| 패턴 | 적용 대상 |
|------|----------|
| 본인 데이터만 (`user_id = auth.uid()`) | brand_voices, keywords, contents, images, channels, schedules, projects, image_scripts, video_scripts, user_prompts |
| 본인 + 관리자 | users, usage_logs, payments, support_tickets |
| 전체 읽기 | plans, brand_voice_presets, keyword_opportunities, industries |
| 관리자만 | admin_prompts, email_logs |

## 3. 데이터 암호화

### AES-256-GCM (저장 시 암호화)
- 대상: 소셜 채널 access_token, refresh_token
- 키 파생: ENCRYPTION_KEY (또는 JWT_SECRET)를 SHA-256 해시
- 저장 형식: `{iv_hex}:{auth_tag_hex}:{ciphertext_hex}`
- 코드 참조: `src/lib/crypto.ts`

### 전송 보안
- HTTPS 전제 (프로덕션)
- 비밀번호 원문 저장 안 함 (bcrypt 해시만)

## 4. 입력 검증

### Zod 스키마 검증

12개 이상의 스키마로 모든 API 입력을 검증한다. 검증 실패 시 `ValidationError(400)`.

주요 스키마: `loginSchema`, `registerSchema`, `keywordAnalyzeSchema`, `contentGenerateSchema`, `imageGenerateSchema`, `brandVoiceCreateSchema`, `scheduleCreateSchema`, `supportTicketCreateSchema`, `profileUpdateSchema`, `channelCreateSchema`, `socialSettingsUpdateSchema`, `adminMailSchema`

코드 참조: `src/lib/validations.ts`

### 환경변수 검증
- Zod로 서버 시작 시 필수 환경변수 검증
- 누락 시 서버 시작 차단
- 코드 참조: `src/lib/env.ts`

## 5. XSS / 인젝션 방지

### HTML Sanitization
`sanitizeInput()` -- `<script>`, `<iframe>`, `<object>`, `<embed>` 태그, `on*` 이벤트 핸들러, `javascript:` URI 제거

### 프롬프트 인젝션 방지
`sanitizePromptInput()` -- 다음 패턴 차단 (`[blocked]`로 치환):
- "ignore previous instructions"
- "ignore above"
- "disregard instructions"
- "forget everything"
- "system prompt"
- 입력 최대 2,000자 제한

### 이메일 HTML 이스케이프
`escapeHtml()` -- `&`, `<`, `>`, `"`, `'` 이스케이프 (사용자 이름을 이메일 HTML에 삽입 시)

### SQL 인젝션 방지
- Supabase 클라이언트 사용 (parameterized queries)

코드 참조: `src/lib/sanitize.ts`, `src/lib/email.ts`

## 6. 요청 제한 (Rate Limiting)

IP + 경로 기반 인메모리 Map. 5분마다 만료 항목 자동 정리. 초과 시 429 + `Retry-After` 헤더.

| 리미터 | 윈도우 | 최대 요청 | 대상 |
|--------|--------|----------|------|
| `loginRateLimit` | 1분 | 10회 | 로그인 |
| `registerRateLimit` | 1분 | 5회 | 회원가입 |
| `forgotPasswordRateLimit` | 1분 | 3회 | 비밀번호 찾기 |
| `passwordChangeRateLimit` | 1분 | 5회 | 비밀번호 변경 |
| `supportRateLimit` | 1분 | 10회 | 고객지원 |
| `channelRateLimit` | 1분 | 10회 | 채널 관리 |
| `socialSettingsRateLimit` | 1분 | 10회 | 소셜 설정 |
| `accountDeleteRateLimit` | 5분 | 3회 | 계정 삭제 |
| `generateRateLimit` | 1분 | 10회 | AI 생성 API |

클라이언트 IP 추출 우선순위: `X-Forwarded-For` > `X-Real-IP` > `127.0.0.1`

코드 참조: `src/lib/rate-limit.ts`

## 7. 사용량 제한

- 플랜별 월간 한도 체크 (`checkUsageLimit`)
- 월초(1일) 자동 리셋
- admin 역할: 모든 제한 해제
- `limit = 0`: 무제한
- 에러 발생 시 안전하게 `allowed: true` 반환 (서비스 차단 방지)
- 초과 시: `USAGE_LIMIT_EXCEEDED` (403)

코드 참조: `src/lib/usage.ts`

## 8. 에러 처리

| 클래스 | HTTP | 코드 |
|--------|------|------|
| `AppError` (기본) | 500 | INTERNAL_ERROR |
| `AuthError` | 401 | AUTH_REQUIRED |
| `ForbiddenError` | 403 | FORBIDDEN |
| `NotFoundError` | 404 | NOT_FOUND |
| `ValidationError` | 400 | VALIDATION_ERROR |
| `RateLimitError` | 429 | RATE_LIMIT_EXCEEDED |
| `UsageLimitError` | 403 | USAGE_LIMIT_EXCEEDED |

- 프로덕션에서 AppError 이외의 에러는 상세 메시지 숨김 ("서버 오류가 발생했습니다")
- 모든 에러 console.error 기록

코드 참조: `src/lib/errors.ts`
