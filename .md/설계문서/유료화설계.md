## 1. 플랜 체계

### 1.1 플랜 종류

| 플랜 | name | 월 요금 | 연 요금(월환산) | 대상 |
|------|------|---------|----------------|------|
| Free | `free` | 0원 | 0원 | 체험/입문 |
| Starter | `starter` | 19,900~29,000원 | 16,600~23,200원 | 소규모 |
| Pro | `pro` | 59,000원 | 47,200원 | 중규모 |
| Business | `business` | 149,000원 | 119,200원 | 대규모/팀 |
| Premium | `premium` | 별도 | 별도 | VIP/무제한 |

> 002_seed.sql과 003_v3_pipeline.sql에 각각 다른 starter 시드가 있어 실제 운영 환경에서는 하나로 통일 필요.

### 1.2 플랜별 한도

| 리소스 | Free | Starter | Pro | Business | Premium |
|--------|------|---------|-----|----------|---------|
| 콘텐츠 생성/월 | 10 | 50~100 | 500 | 무제한(0) | 9,999 |
| 키워드 분석/월 | 20 | 100~200 | 1,000 | 무제한 | 9,999 |
| 이미지 생성/월 | 5 | 30~50 | 200 | 무제한 | 9,999 |
| 저장 키워드 | 20 | 100~200 | 1,000 | 무제한 | 9,999 |
| 채널 연동 | 1 | 3 | 10 | 무제한 | 9,999 |
| 브랜드 보이스 | 1 | 3 | 10 | 무제한 | 9,999 |
| 팀 멤버 | 0 | 0 | 3 | 20 | 10 |
| 이력 보관(일) | 30 | 90~180 | 무제한(0) | 무제한 | 9,999 |
| API 접근 | X | O | X | O | O |
| 우선 지원 | X | X | X | O | O |

**한도 값 규칙:**
- `0` = 무제한 (checkUsageLimit에서 즉시 허용)
- `9999` = 사실상 무제한
- 실수치 = 해당 수만큼 제한

## 2. 사용량 추적

### 2.1 추적 방식

월간 사용량 = `usage_logs` 테이블에서 해당 월(1일~말일)의 action_type별 count

### 2.2 action_type -> 한도 매핑

| action_type | plans 컬럼 | 설명 |
|-------------|-----------|------|
| `content_create` | `content_limit` | 콘텐츠 생성 |
| `keyword_analyze` | `keyword_limit` | 키워드 분석 |
| `image_generate` | `image_limit` | 이미지 생성 |
| `title_generate` | - | 별도 한도 없음 |
| `draft_generate` | - | 별도 한도 없음 |
| `image_script_generate` | - | 별도 한도 없음 |
| `video_script_generate` | - | 별도 한도 없음 |
| `quality_score` | - | 별도 한도 없음 |
| `content_revise` | - | 별도 한도 없음 |
| `publish` | - | 별도 한도 없음 |

### 2.3 checkUsageLimit

- admin 역할: 모든 기능 무제한
- limit 값 0: 무제한
- 에러 발생 시: 안전하게 `allowed: true` 반환 (서비스 차단 방지)
- 초과 시: `USAGE_LIMIT_EXCEEDED` 또는 `PLAN_UPGRADE_REQUIRED` 코드 반환

코드 참조: `src/lib/usage.ts`

## 3. 결제 시스템

### 3.1 결제 세션 생성 (Checkout)

`POST /api/payments/checkout`

1. plan_id로 플랜 조회 (is_active = true)
2. 금액 계산: monthly → `price_monthly`, yearly → `price_yearly * 12`
3. 무료 플랜(amount <= 0) 결제 시 차단
4. 주문 ID: `MF_{userId}_{timestamp}`
5. 결제 세션 정보 반환

### 3.2 결제 승인 (Confirm)

`POST /api/payments/confirm`

1. 서버 사이드 금액 검증 (클라이언트 금액 위변조 방지)
2. 토스페이먼츠 결제 승인 API 호출 (현재 주석 처리, UI만 구현)
3. 만료일 계산: monthly +1개월, yearly +1년
4. `payments` INSERT + `users` UPDATE (plan_id, plan_started_at, plan_expires_at)

### 3.3 토스페이먼츠 연동 (준비됨)

```
POST https://api.tosspayments.com/v1/payments/confirm
Authorization: Basic {base64(TOSS_SECRET_KEY + ':')}
Body: { paymentKey, orderId, amount }
```

현재 주석 처리 상태. 운영 환경에서 활성화 필요.

## 4. 구독 관리

- 조회: `GET /api/subscriptions` -- users.plan_id, plan_started_at, plan_expires_at 기반
- 취소: `POST /api/subscriptions/cancel`

### 업그레이드
즉시 적용: plan_id 변경 + 새 만료일 설정

### 다운그레이드
예약 방식: `pending_plan_id`에 다운그레이드 플랜 저장, `pending_plan_at`에 현재 플랜 만료 시점 설정

## 6. 사용량 초과 처리

### 클라이언트
FlowGuard 컴포넌트가 플랜 만료 시 차단

### 서버
`checkUsageLimit()` → 초과 시 403 응답:
```json
{
  "allowed": false,
  "used": 50, "limit": 50, "remaining": 0,
  "code": "USAGE_LIMIT_EXCEEDED",
  "resetDate": "2026년 4월 1일"
}
```

## 8. 수익 지표 (관리자)

`GET /api/admin/stats`

| 지표 | 계산 |
|------|------|
| MRR | 이번 달 completed 결제 합계 |
| Churn Rate | (전월MRR - 이번달MRR) / 전월MRR * 100 |
| LTV | 전체 매출 / 유료 결제자 수 |
| 플랜 분포 | active 유저의 plan_id별 그룹핑 |
